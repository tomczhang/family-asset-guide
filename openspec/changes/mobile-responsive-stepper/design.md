## Context

当前应用为桌面端设计，使用固定 260px 右侧 TableOfContents 侧边栏 + 760px 最大内容宽度布局。所有四个编辑区块（AssetEditor、AccessEditor、SopEditor、CustomSectionEditor）在同一页面垂直排列。在移动设备上内容被挤压，侧边栏占据大量空间，完全无法使用。

技术栈：React 18 + TypeScript + Vite，纯 CSS（无 CSS 框架），无路由库（单页应用）。最终产物通过 vite-plugin-singlefile 打包为单个 HTML 文件。

## Goals / Non-Goals

**Goals:**
- 移动端（≤768px）下每个编辑区块独占一个"页面"，通过底部导航按钮切换
- 目录栏在移动端默认隐藏，通过浮动按钮打开覆盖层面板
- 桌面端体验完全不变
- 不引入任何新的 npm 依赖

**Non-Goals:**
- 不做平板专用布局（平板使用桌面布局）
- 不做滑动手势切换页面
- 不引入路由库（使用 React 状态管理步骤切换）
- 不做移动端 PDF 预览

## Decisions

### 1. 断点检测方式：CSS 媒体查询 + React 状态

使用 `window.matchMedia('(max-width: 768px)')` 创建一个 `useIsMobile()` hook，在 React 层控制渲染逻辑。同时用 CSS 媒体查询处理纯样式层的响应式。

**理由**: 需要在 JS 层条件渲染不同组件结构（stepper vs 全部展示），纯 CSS 媒体查询无法满足。`matchMedia` 性能优于 `resize` 事件监听。

### 2. 步骤状态管理：提升到 App 层

在 `AppContent` 中新增 `currentStep` 状态（0-3 对应四个区块）。移动端下只渲染当前步骤的编辑器组件，桌面端忽略此状态，渲染全部。

**理由**: 步骤状态影响多个组件（主内容区、底部导航、目录面板），放在 App 层最自然。不需要改 Context，因为这是纯 UI 状态。

### 3. 移动端目录面板：覆盖层 + 浮动按钮

底部右侧放一个浮动按钮（FAB），点击后弹出全屏覆盖层面板（从右侧滑入），列出四个步骤，点击步骤切换 `currentStep` 并关闭面板。

**理由**: 覆盖层比抽屉更简单，不需要处理复杂的半屏交互。浮动按钮是移动端常见模式，用户直觉可达。

### 4. 底部步骤导航栏

移动端底部固定显示"上一步/下一步"按钮栏，同时显示当前步骤指示器（1/4, 2/4 等）。最后一步的"下一步"替换为"生成 PDF"。

**理由**: 给用户明确的线性前进路径，比自由跳转更符合"逐步填写"的心智模型。

### 5. Toolbar 移动端适配

移动端下隐藏"导入草稿""导出草稿""清空数据"按钮文字，只保留图标，或收进一个"更多"菜单。

**理由**: 移动端 toolbar 空间有限，需要精简。

## Risks / Trade-offs

- **步骤切换不保留滚动位置** → 每次切换重置到顶部，可接受因为每步内容不长
- **无路由意味着无法分享特定步骤链接** → 非目标用户场景，应用本身是离线工具
- **768px 断点可能不适合部分设备** → 这是业界标准移动断点，覆盖绝大多数手机
